# QAエンジニア戦略ドキュメント — kakebo（家計簿SaaS）

> 作成日：2026-05-02
> 対象：React + TypeScript + Vite 製の家計簿SaaS「kakebo」
> 前提：4つの請求グループ（PayPay / セゾン / イオン / JCB）、カード→グループ→引落日の連鎖計算、ローン（ボーナス加算）、銀行CSV取込、テスト未整備
> ミッション：**「金額が1円でも合わなかったら不合格」** という金融計算特有の品質基準を全社で徹底する

---

## 0. 全体方針

家計簿SaaSは「ユーザーが信頼してお金を任せる」プロダクトであり、表示上の不一致が即座に「使えない」評価につながる。
QAは「金額の整合性」を最優先のミッションとし、その他のUX/パフォーマンスはその次に位置付ける。

| 観点 | 優先度 | 説明 |
|---|---|---|
| 金額計算の正確性 | S | 1円単位で誤差を許容しない |
| 締め日／引落日ロジック | S | カレンダー仕様の正確性 |
| データ整合性（永続化・移行） | A | localStorage破損／インポート再現性 |
| UX（ユーザビリティ） | B | 使いやすさ・分かりやすさ |
| パフォーマンス | C | 取引数1万件でも操作可能 |

## 1. テストピラミッド設計

```
        ╱╲
       ╱E2E╲       5%   Playwright（クリティカルフロー10〜15シナリオ）
      ╱──────╲
     ╱統合テスト╲   25%  Vitest + Testing Library（ストア／フック／CSV取込）
    ╱──────────╲
   ╱  単体テスト  ╲ 70%  Vitest（金額計算・日付ロジック・純粋関数）
  ╱────────────────╲
```

### 1.1 各レイヤーの責務とツール

| レイヤー | 比率 | 主ツール | 責務 |
|---|---|---|---|
| Unit | **70%** | Vitest | 金額計算関数、締め日計算関数、丸め処理、ローン按分、CSVパーサ。**ここで金額系バグの大半を捕捉する** |
| Integration | **25%** | Vitest + @testing-library/react、msw | Zustand/Redux等のストア更新、カード→グループ→引落日の連鎖、CSV取込→固定費検出フロー、コンポーネント＋フック結合 |
| E2E | **5%** | Playwright | 取引登録→月次集計表示→CSVエクスポート、ブラウザ間／タイムゾーン跨ぎ、localStorage永続化の往復 |

### 1.2 採用理由

- **Vitest**：Vite前提のプロジェクトとネイティブ統合、HMRライクな高速ループでTDDに最適。
- **Testing Library**：実装詳細でなくユーザー視点でのDOM検証を強制。
- **Playwright**：複数ブラウザ・タイムゾーン・CSVダウンロードを扱える。Cypressより並列実行とTZ操作が強い。

### 1.3 カバレッジ目標

| 種別 | 目標 |
|---|---|
| Statements | 80% |
| Branches | 85%（金額・日付ロジックは100%） |
| Functions | 80% |
| 金額計算モジュール | **100%（必須）** |

| ID | Given | When | Then |
|---|---|---|---|
| DI-01 | localStorageに不正JSON（手動編集） | アプリ起動 | 例外で落ちず、バックアップから復元 or 初期化UI表示 |

### 4.1 「金額完全一致」検証スクリプト

エクスポート→インポート間の検証は手動でなく **dump → 数値ハッシュ比較** で自動化する：

```ts
const hash = (data) => sha256(JSON.stringify(canonicalize(data)));
expect(hash(beforeExport)).toBe(hash(afterImport));
```

## 7. 金融計算特有の品質基準

### 7.1 黄金律：「1円でも合わなかったら不合格」

- 表示金額のテストは **toBe(整数値)** で厳密一致。`toBeCloseTo` 禁止。
- スナップショットテストの金額部分は別途 toBe 検証を併記。
- 月次合計 ＝ Σ取引 ＝ Σグループ別 を **3経路一致** で検証。

### 7.2 通貨ライブラリの採用検討

| 候補 | 推奨度 | 理由 |
|---|---|---|
| **dinero.js v2** | ◎ 第一候補 | 不変オブジェクト、JPY最小単位（円）対応、配賦（allocate）で端数を最終回寄せが容易 |
| decimal.js | ○ | 小数演算が必要な手数料・利率計算に併用 |
| 自前 number 演算 | × | float誤差・丸め忘れの温床。即座に廃止する |

**方針**：内部はすべて整数（円単位）で保持し、`Money` ラッパー型を導入。表示時のみフォーマット。

```ts
// Bad
total += transaction.amount * rate;  // float誤差

// Good
total = total.add(transaction.money.multiply(rate, ROUND_HALF_EVEN));
```

### 7.3 不変条件（Invariant）

毎テストの最後に以下が成立することを検証：

1. `sum(transactions) === monthlyTotal`
2. `sum(group.transactions for all groups) === sum(transactions)`
3. `sum(loanInstallments) === loanPrincipal + loanInterest`
4. `exported.then(imported).deepEquals(original)`

### 7.4 プロパティベーステスト（fast-check）

```ts
fc.assert(fc.property(
  fc.array(fc.integer({min: 0, max: 9_999_999})),
  (amounts) => sumMoney(amounts) === amounts.reduce((a,b)=>a+b, 0)
));
```

## 8. テスト自動化のロードマップ

### Phase 1（〜1ヶ月）：手動チェックリスト

- 50ケース回帰リスト（本ドキュメント §6）をスプレッドシート化
- リリース前に手動で全件実行、所要時間を計測
- 発見バグはすべてGitHub Issuesに「テストケース化必須」ラベル

### Phase 2（1〜3ヶ月）：Unit中心

- Vitest導入、金額計算・締め日計算モジュールから着手
- カバレッジ目標：金額モジュール100%、全体50%
- pre-commitフックで影響範囲のテスト実行（lint-staged + vitest related）
- CI（GitHub Actions）でPRごとに Unit + Integration

### Phase 3（3〜6ヶ月）：E2E自動化、CI完全組み込み

- Playwright導入、Phase 1の50ケースのうち40ケースを自動化
- CI matrix：Chrome/Firefox/Safari × JST/PST タイムゾーン
- 毎日深夜にフルE2E（30分許容）、Slack通知
- カバレッジバッジをREADME掲載、80%下回ったらマージブロック

### Phase 4（6ヶ月〜）：継続改善

- 本番ログから「未テストの取引パターン」を抽出して逆流テスト追加
- ミューテーションテスト（Stryker）で「テストの効きの良さ」を計測
- カオステスト：localStorage破損／ネットワーク断のシミュレーション

## 9. バグ発見時のトリアージ基準

### 9.1 Severity（影響度）

| Severity | 定義 | 例 |
|---|---|---|
| **S1 Critical** | 金額が誤って表示される／データロスする | 月次合計が明細合計と1円でも違う、CSV取込で取引消失 |
| **S2 High** | 機能不全だが金額は正しい | 引落予定画面が表示されない、エクスポートできない |
| **S3 Medium** | 機能制限・回避策あり | 一部ブラウザで日付ピッカーが動かない、Safari固有のレイアウト崩れ |
| **S4 Low** | 軽微・見た目 | 金額の桁区切りが一部画面で漏れる（ただし金額自体は正しい） |

### 9.2 Priority（対応優先度）

| Priority | SLA | 説明 |
|---|---|---|
| **P0** | 24時間以内ホットフィックス | リリース停止／撤回も検討 |
| **P1** | 次回リリース必須 | 1〜2週間 |
| **P2** | 3リリース以内 | 1〜2ヶ月 |
| **P3** | バックログ | 3ヶ月以上 |

### 9.3 マトリクス

|  | S1 | S2 | S3 | S4 |
|---|---|---|---|---|
| 全ユーザー影響 | **P0** | P1 | P2 | P3 |
| 一部ユーザー（ブラウザ等） | **P0** | P1 | P2 | P3 |
| 限定条件のみ | **P1** | P2 | P3 | P3 |

> **金額計算系（S1）は条件問わず常に最低 P1、全ユーザー影響なら P0**。これを金科玉条とする。

### 9.5 ポストモーテム

S1 のすべて、S2 の P0/P1 はリリース後にポストモーテム（責任追及でなく再発防止）を実施。
**「なぜテストで捕まえられなかったか」** を必ず分析し、新規テストケースを追加。

---

## 10. 付録：Given/When/Then テストケース管理

### 10.1 すべてのテストケースは以下のフォーマットで記述

```
ID    : CR-20
分類  : 締め日ぴったり
Severity: S1
Given : JCB（締め日15日、引落27日）、ユーザーA
When  : 2026/5/15 14:00 JST に¥3,000を利用登録
Then  : 当期（5月締め）に含まれ、引落予定は2026/6/27と表示
And   : 月次合計に¥3,000が加算される
```

### 10.2 命名規則

- `CR-XX`：Critical Regression（最優先回帰）
- `EG-XX`：Edge Case
- `DI-XX`：Data Integrity
- `US-XX`：Usability Scenario

### 10.3 トレーサビリティ

要件 → テストケース → 自動化スクリプト の3者を一意IDで紐付け、Notionまたはスプレッドシートで管理。

## 11. まとめ

1. **金額の正確性は神聖不可侵**。1円ズレたら即S1。
2. ピラミッドは Unit 70% / Integration 25% / E2E 5%。
3. 通貨は **整数 + dinero.js** で扱い、float禁止。
4. 締め日・引落日・タイムゾーンは「カレンダー仕様書」を別ドキュメント化し、テストはそれに準拠。
5. 50ケース回帰テストを手動 → Vitest → Playwright と段階的に自動化。
6. バグはSeverity×Priorityマトリクスで即時トリアージ、金額系は常時最高優先度。

「使えば使うほどユーザーが安心する」家計簿SaaSを目指し、QAは品質の最後の砦としてこの戦略を遂行する。
