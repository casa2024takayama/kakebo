# kakebo PRD v2 — クレジットカード引落破綻防止アプリ

> 作成日: 2026-05-02 / バージョン: 2.0 (Pivot)
> オーナー: PdM
> ステータス: Draft（社長承認待ち）

---

## 1. プロダクトビジョン

**「来月のカード引落で口座が死なない」を当たり前にする。**

クレジットカード中心の生活者にとって、本当の家計管理は「今月いくら使ったか」ではなく「来月いくら引き落とされるか」を把握することである。kakeboは、複数カードの締め日サイクルとローン・収入を一元的に時系列で見える化し、「翌月引落予定 vs 収入」の差分を毎日のトップ画面に置く。家計簿ではなく、**キャッシュフロー破綻の早期警告システム**として再定義する。

---

## 2. ターゲットユーザーとペルソナ

### プライマリーターゲット
- クレジットカードを生活インフラとして複数枚利用する社会人
- 月収はほぼ固定（給与所得者）、ボーナス年2回
- ローン（住宅・車・カードリボなど）を1件以上抱える
- 紙の家計簿・既存アプリが「カード引落の山」を可視化できず挫折した層

### ペルソナ：社長（ファーストユーザー）
- 40代男性、給与＋ボーナス、住宅ローンあり
- メインカード4系統：PayPay / セゾン / イオン / JCB（＋ニコス・ビックカメラ等のサブカードが上記いずれかに紐付く）
- 痛み：「来月の引落が15日に集中、給料日前に残高が薄い」「ボーナス月の加算ローンを忘れて赤字」
- 行動：銀行アプリとカードアプリを4〜5個ハシゴ。月末に電卓で集計。
- 期待：1画面で「翌月の自分の口座」が見えること。食費分析は不要。

### セカンダリー（Phase 4以降）
- 同じ痛みを持つ20〜50代の生活者全般（SaaS化）

## 3. コアジョブ（Jobs to be Done）

| When | I want to | So I can |
|---|---|---|
| 朝コーヒーを飲みながらアプリを開いた時 | 翌月の引落合計と収入の差分を一目で見たい | 今日カードを切ってよいか判断できる |
| 銀行CSVをダウンロードした時 | 固定費を自動で識別してほしい | 手入力の手間なく予測精度を上げたい |
| ボーナス月が近づいた時 | 加算ローン込みの引落予測を見たい | ボーナスを使い切る前に余力を計算できる |
| 新しいカードを作った時 | 締め日・引落日・所属請求グループを登録したい | 4グループのキャッシュフローに即反映したい |
| 給与・ボーナスが変動した時 | 月収・ボーナスを更新したい | 翌月以降の差分予測に反映したい |

**非ジョブ（やらないこと）**
- 食費が今月いくらかを知る／カテゴリ別に節約のアドバイスを受ける
- レシート1枚ごとの記録（カード明細で代替）

## 4. 機能要件

### Phase 1（自分用MVP・3〜4週間）— トップ画面の作り直し

> **ゴール：社長が毎朝開いて「翌月引落 vs 収入」を確認できる状態。**

#### 1.1 請求グループ管理
- 固定4グループを初期データとして投入：**PayPay / セゾン / イオン / JCB**
- 各グループに `締め日(1〜31 or 月末)` と `引落日(翌月の日付)` を保持
- 編集可能。グループ自体の追加・削除はPhase 1では不要（4固定）

#### 1.2 カード管理
- カード単位で `名前 / 所属請求グループID / カードマスタID(任意)` を持つ
- 例：「ニコスゴールド」→ JCBグループに紐付け、「ビックカメラSuica」→ JCBグループに紐付け
- カード追加時、**カードマスタ**（セゾン/イオン/JCB/三井住友/楽天等）を選ぶと締め日・引落日の初期値が請求グループに提案される（ユーザー上書き可）

#### 1.3 締め日ベース集計（コアロジック）
- 取引データを「請求グループの締め日サイクル」で再集計する関数 `aggregateByBillingCycle(groupId, cycleAnchor)` を新設
- 例：JCBが「15日締め・翌月10日払い」の場合、4/16〜5/15の取引を「2026-06-10引落」として集計
- 月初〜月末のカテゴリ集計ロジックは残すが、ダッシュボードからは外す

#### 1.4 翌月引落 vs 収入ダッシュボード（**新トップ画面**）
- 大見出し：**「翌月の差分： +¥XX,XXX」** （プラス＝黒字、マイナス＝赤字で色分け）
- 内訳テーブル：
  - 収入：月収（固定値）
  - 支出：4請求グループ別の引落予定額（締め日ベースで計算）
  - 支出：ローン月額（Phase 3で追加、Phase 1では手動入力欄で代替）
- 当月＋翌月＋翌々月の3ヶ月タイムラインカード

#### 1.5 取引入力（既存活用）
- 取引に `cardId` フィールドを追加。カード未指定の現金取引も許容
- カード別の取引一覧に絞り込めること

#### 1.6 月収設定
- 設定画面に「月収（手取り）」を1フィールド追加（Phase 1ではボーナスは含めず）

---

### Phase 2（2〜3週間）— 銀行CSVから固定費を自動検出

> **ゴール：手作業の固定費登録を撲滅し、引落予測の網羅性を上げる。**

#### 2.1 銀行CSVインポート（新規）
- 既存のカード明細CSVインポートとは別に「銀行口座CSV」モードを追加
- 摘要・出金額・日付の3列をマッピング

#### 2.2 固定費候補の自動抽出
- 直近3〜6ヶ月の出金から以下の条件でグルーピング：
  - **同一摘要** （正規化：全角半角・末尾の数字列・カナ揺れを丸める）
  - **月1回前後の頻度** （30±7日）
  - **金額が±10%以内に収束**
- カード引落の摘要（「カード」「VISA」「ＶＩＳＡ」「JCB」「クレジット」等のキーワード）は**除外**
- 候補を一覧表示し、ユーザーがチェックして固定費マスタに追加できる

#### 2.3 固定費の引落予測への反映
- 固定費を「銀行から直接引かれる支出」として、ダッシュボードの月次差分計算に組み込む

---

### Phase 3（2〜3週間）— ローン・ボーナスの時系列モデル

> **ゴール：年間キャッシュフローの山谷を見える化。**

#### 3.1 ローン管理
- 項目：`名前 / 月額 / ボーナス加算月（複数選択：6月/12月）/ 加算額 / 開始月 / 終了月（任意）`
- 月次の引落予定に常時加算

#### 3.2 ボーナス収入
- `年2回（6月/12月などユーザー指定） / 各回の手取り見込み額`
- 該当月のダッシュボードでは収入＝月収＋ボーナスとして計算

#### 3.3 12ヶ月キャッシュフロービュー
- 当月〜12ヶ月先までの月次差分を棒グラフで表示
- 赤字月をハイライト

---

### Phase 4（中長期）— SaaS公開

> **ゴール：社長以外のユーザーに開放。**

#### 4.1 マルチテナント化
- ユーザーアカウント（メール＋パスワード or OAuth）
- データを `userId` で分離、サーバーDBへ移行（候補：Supabase / Cloudflare D1）

#### 4.2 認証・認可
- セッション管理、APIキーの暗号化保管

#### 4.3 クラウド同期
- localStorage → サーバーのハイブリッド（オフライン編集、再接続時に同期）

#### 4.4 課金・プラン
- 無料：1請求グループ／有料：無制限＋CSV自動取込（オープンバンキングAPI連携を検討）

## 5. 非機能要件

| 項目 | 要件 |
|---|---|
| パフォーマンス | ダッシュボード初回描画 < 1秒（取引1万件まで） |
| データ保持 | Phase 1〜3：localStorage / Phase 4：サーバーDB＋暗号化 |
| ブラウザ対応 | Chrome / Safari / Firefox 最新版、モバイルレスポンシブ |
| データエクスポート | JSON・CSV出力を全Phaseで維持 |
| プライバシー | API key・取引データは外部送信しない（Phase 1〜3） |
| エラー耐性 | CSV取込失敗時に部分ロールバック、データ破損ゼロ |
| UI言語 | 日本語のみ（Phase 4で多言語検討） |

## 8. 既存実装からの差分

### 残すもの（流用）
- React + TypeScript + Vite + Zustand + Tailwind の技術スタック
- `src/lib/csv.ts`（CSVパース基盤）→ 銀行CSVモードへ拡張
- `src/lib/storage.ts`（localStorage抽象化）→ キー追加で対応
- `src/pages/Transactions.tsx`（取引一覧）→ カード絞り込みを追加
- `src/pages/Import.tsx`（CSVインポートUI）→ モード切替を追加
- `src/pages/AppSettings.tsx`（設定画面）→ 月収・ボーナス・カード管理を追加
- 取引データ構造の `amount / date / memo` フィールド

### 捨てる／降格させるもの
- **トップ画面の「カテゴリ別残り予算」を主役から外す**（Dashboardの全面書き換え）
- カテゴリ別予算機能 → 設定画面の隅に残す程度（コアジョブから外れる）
- レシート撮影入力（AI Vision）→ Phase 1〜3では非アクティブ化。コードは残すが導線から削除
- 「今日使える金額 = 残予算 ÷ 残り日数」表示 → 廃止（カテゴリ予算前提のため）
- 固定費の「毎月1日に自動計上」ロジック → 廃止（Phase 2で締め日ベースの予測に置き換え）

### 新規作成
- **型**：`BillingGroup` / `Card` / `CardMaster` / `Loan` / `Income`
- **ロジック**：
  - `src/lib/billingCycle.ts`（締め日サイクル集計）
  - `src/lib/forecast.ts`（翌月差分・12ヶ月予測）
  - `src/lib/recurringDetector.ts`（固定費自動検出 / Phase 2）
- **ページ**：
  - `Dashboard.tsx` を**全面書き換え**（翌月差分が主役）
  - `BillingGroups.tsx`（請求グループ＆カード管理 / Phase 1）
  - `Loans.tsx`（ローン管理 / Phase 3）
  - `CashflowTimeline.tsx`（12ヶ月ビュー / Phase 3）
- **コンポーネント**：
  - `BillingGroupCard`（4グループの引落予定カード）
  - `MonthDiffHero`（差分のヒーロー表示）
  - `RecurringCandidateList`（固定費候補の承認UI / Phase 2）
- **データ**：`CardMaster` の初期データ（セゾン / イオン / JCB / 三井住友 / 楽天 / アメックス 等）

### 既存型の変更（Breaking）
```ts
// Transaction に cardId を追加
type Transaction = {
  ...
  cardId?: string  // 新規・任意
}
// Settings に月収・ボーナス・請求グループ設定を追加
type Settings = {
  ...
  monthlyIncome: number
  bonuses: { month: number; amount: number }[]  // Phase 3
}
```
- マイグレーション：localStorage既存データに対し `cardId = undefined` で読み込めるよう defensive parsing を実装

## 付録：用語

- **請求グループ**：締め日・引落日を共有するカード群の集計単位（PayPay / セゾン / イオン / JCB の4つ）
- **カードマスタ**：セゾン・イオン等の既知カード会社の締め日デフォルト値辞書
- **締め日サイクル**：「15日締め・翌月10日払い」のような請求の1サイクル
- **翌月差分**：(翌月の収入見込み) − (翌月の引落予定合計) の符号付き金額
